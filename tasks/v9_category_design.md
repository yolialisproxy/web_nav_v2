# 第九次开发：分类重构设计方案

## 一、项目背景与目标

### 1.1 当前状态
- **主数据文件**: `websites.json` (3,342个站点)
- **当前分类体系**: 9个三级分类（实际8个一级分类，但多媒体细分出视频/音频）
- **标准目标分类**: `final_standard_categories.json` (48个叶子分类)
- **原始数据池**: `data/ods_raw_sites.json` (9,903个站点，可用于补充)

### 1.2 核心问题
1. **严重不平衡**: 最大类(AI工具)1,485站 vs 最小类(音频处理)1站
2. **粒度过粗**: 大量站点被归为"XX/XXX/通用"模式
3. **分类覆盖不足**: 只有9个分类，无法满足9,000站点的精细化导航需求

### 1.3 设计目标
- **短期**: 将3,342个站点重构映射到48个叶子分类
- **平衡性**: 每类10-50个站点（当前阶段可适度放宽至5-80个）
- **扩展性**: 为后续补充至9,000站点预留容量
- **语义清晰**: 每个分类有明确的定义和边界

## 四、分类映射方案（核心设计）

### 4.1 映射原则

1. **优先精确匹配**: 根据站点名称、描述、URL语义精确分类
2. **保留"其他"兜底**: 设立"其他/未分类"兜底类，但限制容量≤100
3. **新旧分类兼容**: 支持逐步迁移，避免一次性全部重分类
4. **可追溯性**: 保留原category字段，新增`category_v9`字段

## 五、容量扩展策略

### 5.1 当前容量问题

**现状**:
- AI工具类: 1,485站（严重超载，标准类应≤50站）
- 其他类: 938站（应压缩至≤100站）
- 系统工具: 718站（严重超载）

**瓶颈**: 如果强制将3,342站压入48类，平均每类需69.6站，超出50站上限。

### 5.2 扩容方案（三选一）

#### **方案A：扩展标准分类树（推荐）**

**目标**: 将48个叶子类扩展至60-80个叶子类，确保每类≤50站

**实施步骤**:

1. **增加"AI与智能"主分类**（新增12-15个叶子类）
   - AI工具平台
   - AI模型库
   - AI API服务
   - AI学习资源
   - AI社区论坛
   - AI代码助手
   - AI图像生成
   - AI文本生成
   - AI语音处理
   - AI视频生成
   - AI数据集
   - AI研究论文

2. **增加"多媒体"主分类**（新增8-10个叶子类）
   - 视频平台（抖音、YouTube、B站等）
   - 视频工具（剪辑、制作、特效）
   - 音频平台（Music、Podcast、SoundCloud）
   - 音频工具（编辑、混音、处理）
   - 直播平台
   - 动画制作
   - 素材库（视频/音频素材）

3. **细化"开发工具"分类**（新增5-8个叶子类）
   - 版本控制（Git平台）
   - CI/CD工具
   - 容器与云原生
   - 监控与调试
   - 测试工具
   - 包管理
   - 代码托管

4. **增加"系统与效率"主分类**（新增6-8个叶子类）
   - 任务管理
   - 笔记与知识库
   - 日历与时间管理
   - 密码与安全
   - 文件管理
   - 自动化工具
   - 监控工具

**扩展后分类总数**: 48 + (12+8+5+6) = **79个叶子分类**

**容量验证**: 3,342 ÷ 79 ≈ 42.3站/类，符合10-50站标准

**优势**:
- 语义清晰，分类明确
- 每类站点数合理，便于维护
- 为扩充至9,000站预留空间（9,000÷79≈114，需进一步扩展）

**劣势**:
- 需要修改`final_standard_categories.json`标准文件
- 导航界面层级可能变深

#### **方案B：保持48类，弹性容量（折中）**

**目标**: 保持48个标准叶子类，允许超大类的存在（上限100站）

**实施策略**:
- 允许AI工具类保留80-100站（使用子标签过滤）
- 其他类压缩至50站以内
- 系统工具拆分，保留核心50站

**容量分配**:
```
AI工具: 100站（限流）
设计创意: 200站（4个叶子类 × 50）
开发工具: 500站（10个扩展叶子类 × 50）
效率办公: 200站（5个叶子类 × 50）
多媒体: 250站（5个扩展叶子类 × 50）
... + 其他新增类
```

**优势**:
- 保持标准分类树不变
- 开发工作量小

**劣势**:
- 违背"平衡性"设计原则
- AI类仍然过于臃肿，用户体验差

#### **方案C：混合模式（推荐）**

**目标**: 核心48类 + 扩展类，形成"标准类+主题类"混合体系

**结构**:
```
Level 1: 主分类（原4个 + 新增2个 = 6个）
Level 2: 子分类（原12个 + 新增6个 = 18个）
Level 3: 叶子分类（原48个 + 新增20个 = 68个）
```

**扩展方式**:
- **标准叶子类** (48个): 保持原标准分类，每类≤50站
- **扩展叶子类** (20个): 新增类，专用于分流超大类（如AI工具拆分12个类）

**优势**:
- 兼容原有标准
- 灵活扩容
- 既保持规范性，又解决容量问题

### 7.2 映射脚本设计

```python
# classify_v9.py 核心逻辑
class V9Classifier:
    def __init__(self):
        self.load_mapping_rules()
        self.load_keyword_dict()

    def classify(self, site):
        """分类主函数"""
        # 1. 提取特征
        features = self.extract_features(site)

        # 2. 规则匹配
        matched_rule = self.match_rules(features)
        if matched_rule and matched_rule.confidence > 0.8:
            return matched_rule.target_category

        # 3. 相似度匹配（与现有分类样例对比）
        similar = self.find_similar_sites(site)
        if similar:
            return similar.most_common_category

        # 4. 兜底分类
        return self.default_category(site)

    def extract_features(self, site):
        """提取网站特征"""
        name = (site.get('name', '') + ' ' + site.get('title', '')).lower()
        desc = site.get('description', '').lower()
        url = site.get('url', '').lower()

        features = {
            'has_ai': any(w in name+desc for w in ['ai', 'gpt', 'machine learning']),
            'has_video': any(w in name+desc+url for w in ['video', 'youtube', 'bilibili']),
            'has_design': any(w in name+desc for w in ['icon', 'font', 'ui', 'design']),
            'has_code': any(w in name+desc for w in ['github', 'api', 'code']),
            'has_doc': any(w in name+desc for w in ['docs', 'wiki', 'documentation']),
            'is_platform': any(w in name+desc for w in ['platform', 'cloud', 'service']),
            # ... 更多特征
        }
        return features
```

## 十四、总结与建议

### 14.1 核心建议

1. **采用方案C（混合模式）**: 80个叶子类 + 保留扩展灵活性

- ✅ **容量监控**: 实时监控每类容量，预警机制
- ✅ **人工兜底**: 低置信度站点人工审核，保证质量
